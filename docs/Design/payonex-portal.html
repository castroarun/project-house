03 — Why These Choices

Design Decisions

Every technical choice was weighed against alternatives. Here’s what we picked and why — documented, not hand-waved.

ADR-001
“Which event streaming platform?”

Event Streaming

✓ Apache Kafka over RabbitMQ, AWS SQS
Immutable log = audit trail + replay + event sourcing. Payment systems cannot lose events. Partition-key ordering guarantees per-payment sequencing.
ADR-002
“What architecture pattern for domain isolation?”

Architecture Pattern

✓ Hexagonal + Event Sourcing over Layered MVC
Domain logic is completely isolated from infrastructure. Each region uses different adapters, same core. Ports & adapters make clearing house protocols swappable.
ADR-003
“Separate codebases per region, or one?”

Multi-Region Strategy

✓ Config-Driven Adapters over Per-region forks
One Docker image, one pipeline. Config decides which clearing adapters, compliance rules, and operating hours activate at startup. Add a new country = add one adapter config.
ADR-004
“Sequential compliance or parallel?”

Compliance Engine

✓ Internal Async Saga over Sequential pipeline
4 screening engines run as parallel futures inside Tier 3. Aggregator waits for all, publishes single event. Sub-500ms total instead of 2s sequential.
04 — Concepts

The PayoneX Language

Purpose-built concepts that solve real payment engineering problems. Each one addresses a gap no off-the-shelf pattern fully covers.

Phantom
Clearing House Simulator
A standalone Docker container that simulates every clearing house response — ACSC success, RJCT rejection, timeout, partial settlement, network error. You can’t test against FAST or MEPS+ in dev. Phantom replaces the real clearing network so the full payment lifecycle runs end-to-end without a live connection.
Anti-Corruption Layer
Protocol Translator
Each clearing house speaks a different protocol dialect. ACL adapters isolate their quirks — message format translation, operating hour validation, minimum amount enforcement — so the core domain never knows if it’s talking to Singapore FAST or UK Faster Payments.
Config-Driven Region
One Image, Any Country
Same Docker image processes Singapore, Hong Kong, UK, and India. No fork, no separate deployment. A config file decides which clearing adapters, compliance rules, and operating hours activate at startup. Add a new country = add one adapter config.
Accounting Saga
5-Step Settlement
A 5-step double-entry transaction split across clearing. Part 1 (pre-clearing): debit customer, calculate fees, FX convert. Part 2 (post-clearing): settle Nostro, GL posting. On rejection, all steps auto-reverse in order — zero manual reconciliation.
05 — Under the Hood

Engineering Patterns

Battle-tested patterns chosen for payment processing. Each solves a specific operational problem.

Pattern Why In PayoneX
Hexagonal Architecture Domain never depends on infra Core payment logic is protocol-agnostic — Kafka, REST, or IBM MQ are just adapters
Event Sourcing Every change is an immutable event Full audit trail. Replay any payment from inception. Zero data loss on failure
Domain-Driven Design Bounded contexts with clear ownership 6 services, each owns its database. No shared state, no coupling
Saga Pattern Multi-step transactions, auto-rollback 5-step accounting. If settlement fails, all prior entries auto-reverse
Strategy Pattern Pluggable algorithms at runtime Clearing adapters discovered by config. Add a country = add one adapter
Anti-Corruption Layer Shield domain from external quirks Each clearing house gets its own translator. FAST quirks stay in FAST’s adapter
Circuit Breaker Fail fast, recover automatically Resilience4j: 3 failures → OPEN, auto-retry in 30s → HALF-OPEN
Transactional Outbox Exactly-once event publishing Debezium CDC ensures every event reaches Kafka, even if service crashes mid-write
SOLID + CUPID Clean, testable, composable code SRP, open-closed, dependency inversion — enforced by coding standards
